# TON Wallet (Testnet)

Self-custodial web-кошелёк для TON testnet. Работает полностью в браузере без собственного backend. Мнемоника шифруется паролем (AES-256-GCM) и хранится в localStorage.

## Запуск

```bash
npm install
npm run dev
```

Приложение доступно на `http://localhost:5173`.

Или через Make:

```bash
make install
make build             # собирает dist/ + единый dist/ton-wallet.html
make preview
```

`make build` создаёт `dist/ton-wallet.html` — самодостаточный HTML-файл со встроенными CSS и JS, который можно открыть в браузере без сервера.

### Тесты

```bash
make test              # все тесты (vitest + playwright)
make test-unit         # только юнит/интеграционные (vitest)
make test-unit-watch   # vitest в watch-режиме
make test-e2e          # только E2E (playwright)
```

Покрытие: 130 юнит/интеграционных тестов + 20 E2E тестов.

### Конфигурация (.env)

```bash
cp .env.example .env
```

Параметры в `.env`:

| Переменная | Описание | По умолчанию |
|---|---|---|
| `VITE_TONCENTER_URL` | Endpoint TON Center API | `https://testnet.toncenter.com/api/v2` |
| `VITE_TONCENTER_API_KEY` | API-ключ TON Center | пусто (работает с rate limit) |

Для получения бесплатного testnet API-ключа: `/get testnet` боту [@tonapibot](https://t.me/tonapibot) в Telegram.

Vite подставляет значения `VITE_*` при билде — они хардкодятся в бандл.

## Архитектура

- **@ton/ton**, **@ton/core**, **@ton/crypto** — взаимодействие с блокчейном TON

### Кошелёк

Используется **WalletContractV4** — стандартный контракт кошелька TON. При создании генерируется 24-словная мнемоническая фраза через `@ton/crypto`. Приватный ключ не хранится — вычисляется из мнемоники на лету при подписании транзакций.

### Хранение данных и шифрование

Мнемоника шифруется паролем пользователя и хранится в localStorage:

```
localStorage["ton_wallet"] = {
  encrypted: "base64...",   // AES-256-GCM ciphertext
  salt: "base64...",        // 16 байт, случайный
  iv: "base64...",          // 12 байт, случайный
  address: "0Q..."          // публичный адрес (открытый текст)
}
```

Схема шифрования:
1. Из пароля и случайного salt выводится ключ через **PBKDF2** (SHA-256, 600 000 итераций)
2. Мнемоника шифруется **AES-256-GCM** с выведенным ключом и случайным IV
3. AES-GCM обеспечивает аутентифицированное шифрование — неверный пароль даёт чёткую ошибку

Адрес хранится открытым (он публичный), что позволяет показывать баланс и транзакции до разблокировки.

### Три состояния приложения

| Состояние | Условие | Экран |
|-----------|---------|-------|
| Нет кошелька | localStorage пуст | Welcome → Create / Import |
| Заблокирован | Зашифрованный кошелёк в localStorage, мнемоника не в памяти | UnlockWallet (ввод пароля) |
| Разблокирован | Мнемоника расшифрована и в памяти | Dashboard / Send / Receive |

Кнопка "Заблокировать" стирает мнемонику из памяти без удаления из localStorage. "Выйти из кошелька" удаляет всё.

### Отправка TON

При отправке:
1. Комиссия рассчитывается через `estimateFee` API toncenter — строится реальное подписанное сообщение, сериализуется в BOC, отправляется на оценку
2. При ошибке API — fallback на фиксированную оценку 0.01 TON
3. `fwd_fee` (плата за доставку сообщения) добавляется к value, чтобы получатель получил ровно указанную сумму
4. Все сетевые вызовы обёрнуты в retry (до 3 попыток с нарастающей задержкой) для устойчивости к 429 rate limit
5. Форма и экран подтверждения показывают: сумму, комиссию, итого с баланса

## Защита от подмены адресов (Anti-Spoofing)

### Проблема

Address poisoning — атака, при которой злоумышленник отправляет жертве микротранзакции с адреса, визуально похожего на адрес, с которым жертва часто взаимодействует (совпадают первые и последние символы). Жертва копирует адрес из истории, не замечая подмену.

### Реализованные проверки

**1. Обнаружение dust-транзакций**
- Входящие транзакции < 0.001 TON помечаются предупреждением в истории
- Предупреждение: "Микротранзакция — возможная попытка отравления истории адресов"
- Логика: `isDustTransaction()` в `utils/address.ts`

**2. Обнаружение похожих адресов при отправке**
- Введённый адрес сравнивается со всеми адресами из истории транзакций
- Если найден адрес с совпадающими первыми 6 и последними 6 символами, но отличающийся в середине — критическое предупреждение
- Отличающиеся символы подсвечиваются для наглядного сравнения
- Отправка возможна только после явного подтверждения чекбоксом
- Логика: `addressesAreSimilar()`, `checkAddressSpoofing()` в `utils/address.ts`

**3. Верификация буфера обмена**
- При вставке адреса сохраняется оригинальное значение
- На экране подтверждения проверяется совпадение с вставленным
- Если не совпадает — предупреждение о возможном перехвате буфера (clipboard hijacking)
- Если совпадает — информационное напоминание проверить адрес

**4. Визуальное выделение адреса**
- Адрес разбивается на три сегмента с цветовой кодировкой: начало, середина, конец
- Помогает быстро сверить ключевые части адреса

### UX-решения для предупреждений

- Критические предупреждения (похожий адрес) — красная рамка
- Отправка при наличии предупреждений требует чекбокса "Я проверил адрес"
- Кнопка отправки меняет цвет на красный
- Dust-транзакции выделяются оранжевой рамкой в списке

## Выбранные компромиссы

| Решение | Альтернатива | Почему выбрано |
|---------|-------------|----------------|
| AES-256-GCM + PBKDF2 через Web Crypto API | Хранение без шифрования / внешние библиотеки | Нативный браузерный API, нет зависимостей, 600K итераций PBKDF2 делают brute-force дорогим |
| localStorage для хранения | IndexedDB / sessionStorage | Простота API, персистентность между сессиями, достаточно для одного JSON-объекта |
| Адрес хранится открытым | Шифровать всё | Адрес публичный (виден в блокчейне), открытый адрес позволяет показывать баланс до разблокировки |
| Нет backend | Backend для хранения ключей / прокси API | Требование ТЗ. Все запросы напрямую к TON Center API |
| API-ключ в коде | Переменные окружения / backend прокси | Без backend негде спрятать ключ; testnet ключ не имеет ценности |
| HashRouter | BrowserRouter | Работает как статический SPA и через file:// без серверного роутинга |
| Polling каждые 30 сек | WebSocket / SSE | TON Center v2 не предоставляет WebSocket; polling прост и достаточен для кошелька |
| Предварительная estimateFee через POST API | Фиксированная оценка / локальный расчёт | Точная оценка от сети; fallback на 0.01 TON при ошибке |
| fwd_fee добавляется к value | Отправлять value as-is | Получатель получает ровно указанную сумму, а не сумму минус комиссия доставки |
| Retry с backoff при 429 | Показать ошибку сразу | Бесплатный API часто возвращает 429; retry решает проблему прозрачно для пользователя |
| WalletContractV4 | V3 / V5 | Стандартный и наиболее распространённый контракт, совместим со всеми TON-кошельками |
| Подтверждение 3 слов мнемоники при создании | Подтверждение всех 24 / ввод пароля | Баланс между безопасностью и UX |
| Мин. длина пароля 8 символов | Сложные правила (цифры, спецсимволы) | Простота UX; PBKDF2 с 600K итерациями компенсирует слабые пароли |
| Единый HTML-файл (ton-wallet.html) | Стандартная раздача статики | Удобно для распространения и оффлайн-использования |
| vite-plugin-node-polyfills | Ручной polyfill Buffer | Автоматически решает все Node.js polyfill-проблемы для @ton/* библиотек |
